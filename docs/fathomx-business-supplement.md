# FathomX 业务层补充 — 对 Lapis 架构的增量输入

本文档是对 Lapis Research Agent 产品文档的业务层补充。Lapis 定义了清晰的三层技术架构，但没有定义具体的研究方法论。以下内容来自 FathomX 在实际调研中验证过的业务规则，按 Lapis 架构层级组织。

你作为审计者，需要判断每一条是否采纳、修改或拒绝。

---

## Layer 1 补充：Orchestration Layer

Lapis §6.1 定义了 Orchestration Layer 的职责列表。以下是需要增加的具体业务能力。

---

### 补充 1：复杂度前置分类

Lapis §8 定义了"广度优先 / 深度优先 / 平衡"三种 Agent 分配模式，但没有说明如何选择。建议在 Orchestration Layer 增加一个前置分类步骤，根据用户请求的性质决定研究深度。

分类标准：

| 级别 | 触发条件 | 说明 |
|------|---------|------|
| Quick | 用户问的是单一事实、定义、简单查询 | 不调用 Rust MCP，Skill 直接用搜索工具回答 |
| Standard | 用户需要对比、选型、评估、多面分析 | 调用 `research_plan` + 3-4 个 `aspect_research`，覆盖 4 个维度 |
| Deep | 用户需要产业分析、竞品图谱、市场格局、完整调研 | 调用 `research_plan` + 6 个 `aspect_research` + `compare_reports`，覆盖全部 6 个维度 |

Quick 级别是一个重要的"短路"——避免对简单问题启动整个多 Agent 编排流程。Lapis 当前假设所有请求都会走 `deep_research`，但实际使用中大量请求只需要一次搜索就能回答。

建议在 `research_plan` 的返回结构中包含一个 `tier` 字段，让 Orchestration Layer 据此决定后续调用深度。

---

### 补充 2：决策意图推断

在拆解研究任务之前，Orchestration Layer 应先推断用户的"决策意图"——即用户会基于这次调研做什么决策。

这一步改变了整个研究的方向。没有决策意图，Agent 会产出泛泛的信息罗列；有了决策意图，每个 Agent 知道自己的分析最终服务于什么决策，输出变成可行动的判断。

示例：

| 用户问题 | 推断的决策意图 |
|---------|---------------|
| "调研天天跳绳 APP 的用户人群，给出 AI 时代下的方向" | 决定为天天跳绳开发哪些 AI 功能 |
| "对比 Next.js 和 Nuxt.js" | 为新项目选择前端框架 |
| "调研中国智能健身镜市场" | 判断是否进入该市场 / 制定进入策略 |

决策意图应该写入 `research_plan` 的输出中，并传递给每个 Reasoning Layer Agent 作为上下文。

### 补充 3：MECE 6 维度范围扩展

Lapis §6.1 说"将研究任务拆分为多个分析方面"，但没有给出具体的拆分框架。建议采用 MECE（互斥全覆盖）6 维度模板作为默认拆分方式。

6 个维度定义如下：

**维度 1：市场环境 (Market Context)**

调查 TAM/SAM/SOM 市场规模、增长率和增长驱动力、行业生命周期阶段、监管环境和政策方向、宏观经济趋势对该领域的影响。目标是建立"这个市场有多大、往哪走"的全局认知。

**维度 2：竞争格局 (Competitive Landscape)**

识别直接竞品、间接竞品、替代品和潜在进入者。分析每个主要玩家的战略定位、核心优势和弱点。寻找市场空白和竞争动态变化趋势。

**维度 3：用户需求 (User Jobs & Needs)**

用 Jobs-to-be-Done 框架分析用户"雇佣"产品完成什么任务。识别核心任务和支持任务。评估每个任务的结果期望、满意度和机会分数。发现未被服务的用户群和痛点。

**维度 4：产品能力 (Product Capabilities)**

跨竞品的功能对比矩阵，按买家关注的维度打分（0=缺失，1=基础，2=有竞争力，3=最佳）。分析定价架构、技术栈选择和集成生态。

**维度 5：战略定位 (Strategic Position)**

分析每个主要玩家的竞争护城河（网络效应、转换成本、数据壁垒、品牌、监管）。通过 SWOT 识别优劣势和机会威胁。用 ERRC 网格（消除 / 减少 / 提升 / 创造）寻找差异化路径。

**维度 6：未来趋势 (Future Trajectory)**

识别将在 2-3 年内重塑该市场的技术和趋势。分析目前的非客户群体——谁还没使用这类产品、为什么、什么变化可能让他们成为客户。预测市场格局演变。

**使用规则**：
- Quick 级别：只覆盖用户问题最直接相关的 1-2 个维度
- Standard 级别：至少覆盖 4 个维度
- Deep 级别：必须覆盖全部 6 个维度，每个维度有多个子问题

**与 Lapis §6.2 的关系**：Lapis 列出了 9 种可能的 Agent 类型（市场规模、用户痛点、竞品分析等），这些和 MECE 6 维度基本对应但不够结构化。建议将 Lapis 的 9 种 Agent 类型替换为 MECE 6 维度，每个维度对应一个或多个 Agent。

### 补充 4：Gap 驱动迭代

Lapis §6.1 提到 Orchestration Layer 要"识别报告之间的冲突、重复、证据不足和开放问题"，但没有给出具体检查标准。建议增加一套明确的 Gap 检测清单，在收到所有 Agent 的 aspect report 后执行。

检查清单（6 项）：

**检查 1：来源数量**
每个维度是否有至少 3 个独立来源？如果少于 3 个，说明该维度搜索不充分，需要补充搜索。

**检查 2：来源多样性**
每个维度是否有至少 2 种不同类型的来源？比如不能全部来自同一个搜索引擎。如果只用了 Grok 没用 Exa（或反过来），应该用另一个工具补充。

**检查 3：矛盾解决**
不同来源之间的矛盾声明是否已经被标注并呈现双方观点？如果存在未解决的矛盾，需要补充搜索来仲裁。

**检查 4：人格覆盖**
每个适用的研究人格是否已对其负责的维度给出了分析？如果市场维度缺少 Market Analyst 的视角，需要补做。

**检查 5：事实基础**
关键声明是否有数值证据？不能只有定性描述（如"市场很大"），需要有量化数据（如"市场规模 $15.2B"）。如果关键声明缺少数字，需要定向搜索统计数据。

**检查 6：时效性**
市场和竞品相关的数据是否在最近 12 个月内？如果数据过旧，需要加日期过滤条件重新搜索。

**迭代规则**：
- Standard 级别：最多 1 轮 Gap 补充
- Deep 级别：最多 2 轮 Gap 补充
- 如果某个维度在 2 轮后仍不满足某项检查，在最终报告的"限制"章节中明确标注，不继续搜索

**与 Lapis 的关系**：这个机制应同时存在于两个层级。Layer 2 的 Agent 内部 loop 中，judge 应在每轮结束后检查单个 aspect 的 gap 状况，决定是否继续搜索。Layer 1 在收到所有 aspect report 后做跨维度的全局 gap 检查，决定是否调用补充的 `aspect_research`。

### 补充 5：来源可信度评级

Lapis §13 提到证据条目应包含"置信度"，但没有给出评级标准。建议采用 A-E 五级来源可信度评级：

| 等级 | 来源类型 | 典型示例 |
|------|---------|---------|
| A — 权威 | 学术期刊、官方统计、政府发布 | Nature, PubMed, 国家统计局, SEC Filing |
| B — 可靠 | 权威媒体、行业分析机构 | Reuters, Bloomberg, Gartner, McKinsey |
| C — 专业 | 专业媒体、公司官方信息 | TechCrunch, 36Kr, 产品官网, 公司博客 |
| D — 社区 | 社区讨论、用户评论、论坛 | Reddit, 知乎, HackerNews, App Store 评论 |
| E — 不确定 | 来源不明、AI 生成内容、无法追溯 | 无明确出处的信息 |

评级方式：搜索结果返回时，根据 URL 域名做初步分类；最终评级由生成报告的 LLM 根据上下文确定。

最终报告的来源表中，每条来源后面标注其等级，例如：`[3] TechCrunch: AI Fitness Market Report (2026-04) — C`

### 补充 7：优雅降级

Lapis 没有定义当部分组件不可用时系统如何工作。建议增加降级策略：

| 条件 | 行为 |
|------|------|
| Rust MCP 正常 + 所有 Provider 可用 | 全功能：多 Agent 并行搜索分析 |
| Rust MCP 正常但部分 Provider 不可用 | 用可用 Provider 工作，在方法论章节标注覆盖限制 |
| Rust MCP 不可用 | Orchestration Layer 退化为 Claude-only 模式：Claude 直接调用搜索 MCP 工具，自己完成分析，仍然使用 MECE 方法论和报告模板 |
| Rust MCP 中某个 Agent 超时或失败 | 返回已完成的 aspect report，将失败的 aspect 标记为 gap，让 Orchestration Layer 决定是否重试或跳过 |

Claude-only 模式不是产品失败——它仍然比没有方法论的普通搜索好得多。方法论增强（MECE 扩展、决策意图、Gap 检测、报告模板）是纯 prompt 层面的能力，不依赖 Rust MCP。

### 补充 8：三研究人格

Lapis §6.2 列出了 9 种可能的 Agent 类型，但没有给出每种 Agent 的具体指令。建议定义 3 个核心研究人格，每个人格负责 2 个 MECE 维度，携带明确的分析框架。

**人格 1：Market Analyst**

负责维度：市场环境 + 未来趋势

```
你是一位资深市场分析师，专注于市场规模估算、行业动态和增长预测。

你使用的分析框架：
- TAM/SAM/SOM 估算：同时采用自上而下和自下而上两种方法交叉验证
- 市场增长驱动力与抑制因素分析
- 行业生命周期阶段评估
- 监管和宏观经济影响评估

你的输出必须包含：
1. 市场规模估算，附带方法论说明（数据来源、计算过程）
2. 增长轨迹，附带支撑证据
3. 关键趋势和拐点
4. 风险因素和市场不确定性

每个声明必须引用来源材料。低置信度的估算必须明确标注。
如果找到的数据相互矛盾，呈现所有版本并注明差异原因。
如果某个关键数据点缺失，主动搜索而不是跳过。
```

**人格 2：Competitive Intelligence Analyst**

负责维度：竞争格局 + 产品能力

```
你是一位竞争情报分析师，专注于竞品定位、战略动向和市场格局。

你使用的分析框架：
- Porter 五力模型（竞争强度、新进入者威胁、替代品威胁、供应商议价力、买方议价力）
- 竞品功能矩阵：每个功能按 0-3 评分（0=缺失，1=基础，2=有竞争力，3=同类最佳），每个功能带买家重要性权重
- 战略分组映射：在最相关的两个维度上绘制 2x2 定位图
- 护城河分析：逐一评估网络效应、转换成本、数据壁垒、品牌认知、监管壁垒

你的输出必须包含：
1. 竞品识别，分 4 层：直接竞品、间接竞品、替代品、潜在进入者
2. 功能对比矩阵，每列带买家重要性权重和加权总分
3. 竞争定位图（标注选用的两个维度及原因）
4. 每个主要玩家的护城河评估

每个声明必须引用来源材料。信息缺口必须明确标注。
在搜索时优先使用公司研究类工具获取竞品信息。
```

**人格 3：Product Strategist**

负责维度：用户需求 + 战略定位

```
你是一位产品策略师，专注于用户需求分析、产品市场匹配和战略建议。

你使用的分析框架：
- Jobs-to-be-Done：任务地图（核心任务 + 支持任务）、结果期望、机会评分（重要性 + (重要性 - 满意度)）
- Blue Ocean ERRC 网格：针对目标产品，分析应该消除什么、减少什么、提升什么、创造什么
- Opportunity Solution Tree：从目标结果 → 机会 → 解决方案 → 验证实验
- 转换触发器分析：什么事件导致用户离开现有产品 / 采用新产品

你的输出必须包含：
1. 核心任务和支持任务识别
2. 机会评分表：每个关键结果的重要性和满意度（1-10），以及计算出的机会分
3. ERRC 网格，每个象限有具体建议
4. 优先排序的战略建议，每条建议追溯到证据
5. 建议的验证实验

每条建议必须追溯到来源材料中的证据。区分有证据支撑的洞察和推理性判断。
```

**人格与维度的分配关系**：

| 维度 | 主要人格 | 补充人格 |
|------|---------|---------|
| 市场环境 | Market Analyst | — |
| 竞争格局 | CI Analyst | — |
| 用户需求 | Product Strategist | — |
| 产品能力 | CI Analyst | Product Strategist |
| 战略定位 | Product Strategist | CI Analyst |
| 未来趋势 | Market Analyst | Product Strategist |

在 Deep 级别调研中，每个人格对应一个独立的 Reasoning Layer Agent，使用上述指令作为 system prompt。Agent 在自己负责的维度内自主搜索和分析，不需要等别人搜完再开始。

在 Standard 级别调研中，不启动独立 Agent。Orchestration Layer 在最终综合时，按上述三个视角分别审视搜索结果。

### 补充 9：搜索策略指引

Lapis §6.3 定义了 Retrieval Layer 的两个初始 Provider（Exa 和 Grok），但没有指导 Agent 如何选择搜索策略。建议为每个维度提供搜索偏好指引：

| 维度 | 推荐搜索策略 | 原因 |
|------|-------------|------|
| 市场环境 | Grok（AI 搜索 + 额外来源）为主 | 市场报告和分析师数据通常能被通用搜索找到 |
| 竞争格局 | Exa 公司研究为主 + Exa 语义搜索 | Exa 的公司研究能力是获取竞品信息的最佳工具 |
| 用户需求 | Grok + Exa 语义（面向论坛和评论） | 用户评论和讨论散布在多个平台 |
| 产品能力 | Exa 语义 + Grok 网页抓取（产品文档） | 产品功能对比需要抓取官网和文档 |
| 战略定位 | Grok + Exa（相似公司发现） | 分析师报告和战略文章 |
| 未来趋势 | Grok + Exa 语义搜索 | 趋势报告、技术预测、研究论文 |

这个指引应该作为 Agent 的 search planning prompt 的一部分，而不是硬编码的路由规则。Agent 可以根据实际搜索结果自主调整。

```json
  },
  "search_iterations": 2,
  "sources_used": [
    {
      "title": "...",
      "url": "...",
      "provider": "exa",
      "credibility": "B",
      "date": "2026-04"
    }
  ]
}
```

新增字段说明：
- `dimension`：对应 MECE 6 维度中的哪一个
- `persona`：使用了哪个研究人格
- `decision_intent`：从 Orchestration Layer 传入的决策意图
- `gap_status`：5 项 Gap 检查的通过状态
- `search_iterations`：执行了多少轮搜索
- `sources_used`：使用的来源及其可信度评级

### 补充 11：压缩规则

当所有 aspect report 的内容总量超过模型上下文限制时，需要在最终综合前做压缩。压缩不是简单截断，而是有规则的信息密度提升。

压缩规则：

1. **必须保留**：因果关系链、来源之间的矛盾、跨来源的交叉印证、精确数字和统计、来源引用标记
2. **应该减少**：重复印证（改为"经 N 个来源确认"而不是重复描述）、冗长的解释性文字
3. **可以去除**：与主题无关的旁支、低置信度且无佐证的声明、填充性文字
4. **目标**：压缩到原文的约 55%，保留率不低于 35%

压缩后的输出结构：
- **关键事实**：编号列表，每条附来源
- **矛盾点**：来源不一致的地方，呈现双方
- **跨维度主题**：多个维度共同指向的模式
- **信息缺口**：仍然未知或来源不足的点
- **战略信号**：高置信度、应锚定最终报告的关键洞察

## Lapis 开放问题回答

以下是对 Lapis §19 每个开放问题的回答。

**问题 1：Orchestration Layer 的任务拆分规则是否需要配置化？**

不需要。任务拆分规则是产品方法论，不是用户偏好。MECE 6 维度框架应该版本化管理，由产品团队迭代，不暴露为用户可修改的配置项。用户可以影响的是复杂度级别（比如显式要求 Deep 级调研），但拆分逻辑本身不应该配置化。

**问题 2：MVP 阶段是否需要支持用户选择 model provider 和 search provider？**

需要支持用户选择 model provider（按 tier 或 capability 选择），不需要支持用户选择 search provider。

用户关心的是"谁来做我的分析"——这是成本和质量的 tradeoff，用户有正当理由选择。用户不关心"这次搜索用 Exa 还是 Grok"——那是系统内部优化。

具体来说：
- 暴露给用户：model provider 选择（可以按能力类型分：廉价提取、强推理、搜索增强等）
- 不暴露给用户：search provider 路由（系统自动 dispatch）
- 暴露给用户：search provider 的启用/禁用（"我没有 Exa key 所以不用 Exa"）

**问题 3：模型 Agent 的 provider 选择是固定还是动态？**

MVP 先固定。按能力类型分配默认模型，允许用户在配置中覆盖。

动态路由（按 context 长度、分析类型、成本约束自动选模型）是好的方向，但初期数据不足以做好。建议在积累 100+ 次真实调研的使用数据后，根据不同 model × aspect type 组合的质量数据来实现动态路由。

**问题 4：Provider 配置文件应采用 TOML、YAML 还是 JSON？**

TOML。Rust 生态惯例，比 JSON 对人类更友好（支持注释和多行字符串），比 YAML 更安全（无隐式类型转换）。

**问题 5：是否需要将搜索 query 和 source trace 暴露给最终用户？**

需要。放在最终报告的"方法论"章节中。用户需要知道结论是怎么来的，这对建立信任至关重要。

不需要暴露每一条原始 query（太噪），而是汇总统计：用了哪些搜索工具、各搜了多少次、总来源数、去重后来源数、Gap 迭代了几轮。

**问题 6：最终报告是否需要支持多种模板？**

必须从第一天就支持。至少 3 种（Quick / Standard / Deep），对应不同的复杂度级别。模板的差异不只是长度——结构和分析框架完全不同。参见上文"补充 6：报告模板"的完整定义。

**问题 7：是否需要对来源进行自动可信度评分？**

需要。用基于规则的 URL 域名分类做初步评级，最终由 LLM 在报告生成时确定 A-E 等级。不需要 ML 模型——基于 prompt 的启发式方法在实际使用中 90%+ 场景下评级合理。参见上文"补充 5：来源可信度评级"。

**问题 8：是否需要在 Rust 侧保存完整运行记录，还是只返回给 Claude Code？**

返回结构化数据给 Orchestration Layer，由它决定是否持久化到磁盘。

Rust 应返回：aspect report（JSON）、搜索 trace 摘要、错误记录、session 状态。Orchestration Layer 决定哪些写到 workspace 文件（给用户看），哪些只在内存中用于综合。

例外：对于可能超时的长时间 `deep_research`，Rust 应返回足够的中间状态信息，让 Orchestration Layer 能写一个可恢复的 session state，支持断点续研。但持久化的决定权始终在 Orchestration Layer，不在 Rust 侧。
